# 專案 Prompt Engineering 指引

> **用途**：定義專案特定的 AI 行為指引，讓 agent 輸出更貼合領域需求
> **所有區塊皆為可選**：未填寫的區塊，agent 使用內建預設值
> **填寫時機**：`adf.design` 初始化專案時選擇 preset，或任何時候手動編輯
> **大小建議**：每個 agent 範例建議 50-100 行以內，避免佔用過多 context window
>
> **Preset**：`desktop` — 桌面軟體

---

## 🧠 角色補充

- 產品為跨平台桌面應用（Windows / macOS / Linux），需統一使用者體驗
- 啟動速度是關鍵指標，冷啟動 ≤ 3 秒、視窗顯示 ≤ 1 秒
- UI 互動回應時間 < 100ms，長時間操作必須有進度回饋與取消機制
- 安裝/更新體驗必須無痛，支援靜默更新與差量更新
- 使用者資料（設定檔、憑證、本地快取）安全存放，敏感資料加密儲存
- 離線可用是基本要求，網路中斷不應導致 crash 或資料遺失
- 原生系統整合（通知中心、系統匣、檔案關聯）需符合各平台慣例
- 記憶體佔用應控制在合理範圍，避免成為「記憶體怪獸」
- 無障礙（Accessibility）支援不可忽視，需相容螢幕閱讀器

## 📏 評估框架

> `/planner`（architect）比較方案時使用的評估維度和評分標準。
> **覆寫規則**：此處定義的維度**取代**內建預設（不是疊加）。

### 評估維度

| 維度 | 評估重點 | 1 分（差） | 5 分（優） |
|------|----------|-----------|-----------|
| 使用者體驗 | 啟動速度/UI 回應/操作流暢度 | 啟動 > 5 秒、操作卡頓明顯、無載入回饋 | 冷啟動 < 3 秒、所有互動 < 100ms、流暢動畫 |
| 跨平台性 | Win/Mac/Linux 一致性與原生感 | 僅支援單平台或三平台體驗差異極大 | 三平台功能完整、遵循各平台設計規範 |
| 可維護性 | 程式碼結構/測試覆蓋/可除錯性 | 平台特定代碼散落、無測試、crash 難追蹤 | 平台層隔離清晰、UI 可自動測試、完整 crash 報告 |
| 效能 | CPU/記憶體/磁碟使用效率 | 閒置 CPU > 5%、記憶體持續增長 | 閒置 CPU ≈ 0%、記憶體穩定、智慧快取策略 |
| 安全性 | 資料保護/更新機制/輸入驗證 | 憑證明文儲存、無簽章驗證 | OS keychain 整合、程式碼簽章、自動安全更新 |

### 權重規則

- **MVP 階段**：使用者體驗 35%、跨平台性 15%、可維護性 25%、效能 15%、安全性 10%
- **Growth 階段**：使用者體驗 25%、跨平台性 25%、可維護性 20%、效能 15%、安全性 15%
- **Enterprise 階段**：使用者體驗 20%、跨平台性 20%、可維護性 15%、效能 15%、安全性 30%

## 🔍 自我審查

> 各 agent 在輸出前的自我檢查問題（疊加到內建預設之上）。

### 架構設計（architect）

1. 方案是否能在 Windows/macOS/Linux 三平台保持一致的使用者體驗？
2. 啟動路徑是否做到 lazy loading？首屏渲染是否在 1 秒內完成？
3. 離線場景下功能降級策略是否明確？資料同步衝突如何解決？
4. 更新機制是否支援差量更新和失敗回滾？

### 開發實施（developer）

1. 平台特定程式碼是否透過抽象層隔離？是否避免了 `#ifdef` 散落在業務邏輯中？
2. 長時間操作（> 200ms）是否移至背景執行緒？UI 執行緒是否保持無阻塞？
3. 使用者敏感資料是否使用 OS 提供的安全儲存（Keychain/Credential Manager）？

### 代碼審查（code-reviewer）

1. UI 元件是否有適當的 loading/error/empty 狀態處理？
2. 是否有可能導致 UI 凍結的同步呼叫或阻塞等待？
3. 視窗大小調整、多螢幕、高 DPI 場景是否正常運作？

## 📝 範例

> 各 agent 的 few-shot example，展示期望的輸出深度和格式。
> **啟用規則**：有定義時自動啟用 few-shot 模式。

### architect 範例

> **需求**：在視訊會議桌面客戶端中新增「虛擬背景」功能，支援 Windows/macOS/Linux，需於 720p@30fps 即時替換背景，CPU 佔用增量 ≤ 15%。

**硬約束**：
- 三平台功能一致，UI 風格遵循各平台設計規範
- 即時預覽延遲 ≤ 50ms（使用者可感知閾值）
- CPU 增量 ≤ 15%（避免影響視訊編碼品質）
- 安裝包增量 ≤ 20MB

**軟約束**：
- 支援自訂圖片上傳（JPG/PNG）
- 未來擴充：背景模糊（高斯模糊等級可調）
- 支援 GPU 加速（有 GPU 時啟用）

#### 方案 A：ONNX Runtime 推理 + 自研合成

使用預訓練人像分割模型（如 MediaPipe Selfie Segmentation）透過 ONNX Runtime 推理，自研前景/背景合成模組。

- 模型：量化 INT8 模型約 3MB，推理 ≤ 15ms/frame（CPU）
- 合成：SIMD 優化的 alpha blending，≤ 2ms/frame
- 跨平台：ONNX Runtime 支援 Win/Mac/Linux，統一推理介面

#### 方案 B：平台原生 API

Windows 使用 Windows.AI.MachineLearning，macOS 使用 Core ML + Vision Framework，Linux 使用 TensorFlow Lite。

- 效能：各平台原生最佳化，GPU 加速自動啟用
- 合成：各平台分別使用 Direct2D / Core Image / OpenGL
- 跨平台：三套實作，需各自維護

#### 結構化評估

| 維度 | 方案 A（ONNX 統一） | 方案 B（平台原生） |
|------|:--:|:--:|
| 使用者體驗 | 4 — 效能達標、三平台一致 | 4 — 原生加速體驗略優 |
| 跨平台性 | 5 — 單一程式碼庫 | 2 — 三套實作差異大 |
| 可維護性 | 4 — 模型更新一次到位 | 2 — 三平台分別除錯 |
| 效能 | 3 — CPU 推理可達標但餘量小 | 5 — 原生 GPU 加速表現最佳 |
| 安全性 | 4 — 模型本地推理、無雲端依賴 | 4 — 模型本地推理、無雲端依賴 |

**Growth 階段權重加權**：A = 3.75 / B = 3.15

#### 推薦

**推薦方案 A（ONNX Runtime 統一）**，理由：

1. 跨平台一致性是桌面軟體的生命線 — 方案 B 維護三套推理 + 三套合成成本過高
2. CPU 推理 15ms/frame 滿足硬約束，未來可透過 ONNX EP 接入 GPU 提升效能
3. 模型更新（更換更好的分割模型）只需替換 .onnx 檔案，不需改動程式碼
4. 安裝包增量約 15MB（ONNX Runtime 12MB + 模型 3MB），在限制範圍內

**降級策略**：低階硬體偵測到 CPU 不足時，自動降為 480p 推理 + 上採樣。

#### 實施計劃

| 階段 | 任務 | 風險 | 產出 |
|------|------|------|------|
| P1 | ONNX Runtime 三平台整合 + 模型推理驗證 | 🟡 MEDIUM | 推理 benchmark 報告 |
| P2 | 前景分割 + alpha blending 合成模組 | 🟢 LOW | 合成效果 demo |
| P3 | UI 整合（背景選擇器 + 即時預覽） | 🟡 MEDIUM | 功能可用版 |
| P4 | 三平台 UI 驗證 + 效能調優 | 🟡 MEDIUM | 效能報告 |
| P5 | 自訂圖片上傳 + 背景模糊擴充 | 🟢 LOW | 完整功能 |
