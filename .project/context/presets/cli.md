# 專案 Prompt Engineering 指引

> **用途**：定義專案特定的 AI 行為指引，讓 agent 輸出更貼合領域需求
> **所有區塊皆為可選**：未填寫的區塊，agent 使用內建預設值
> **填寫時機**：`adf.design` 初始化專案時選擇 preset，或任何時候手動編輯
> **大小建議**：每個 agent 範例建議 50-100 行以內，避免佔用過多 context window
>
> **Preset**：`cli` — CLI 工具

---

## 🧠 角色補充

- 產品為開發者使用的命令列工具，使用者體驗體現在回饋品質而非 UI 美觀
- 輸入驗證和錯誤訊息是第一印象 — 錯誤訊息必須告訴使用者「哪裡錯了」和「怎麼修」
- 跨平台相容（Windows/macOS/Linux）是基本要求，路徑分隔符號和 shell 差異必須處理
- pipe 友好是核心設計原則：stdout 給機器讀、stderr 給人讀、支援 `--json` 輸出
- 啟動速度至關重要，冷啟動 ≤ 200ms，避免不必要的初始化
- 命令結構遵循 POSIX 慣例：子命令、長/短旗標、`--help` 和 `--version`
- 退出碼（exit code）語義明確：0 成功、1 一般錯誤、2 使用方式錯誤
- 設定檔支援多層級覆寫：內建預設 → 全域設定 → 專案設定 → 命令列參數
- 互動模式與非互動模式都要支援，CI 環境下自動偵測並禁用互動提示

## 📏 評估框架

> `/planner`（architect）比較方案時使用的評估維度和評分標準。
> **覆寫規則**：此處定義的維度**取代**內建預設（不是疊加）。

### 評估維度

| 維度 | 評估重點 | 1 分（差） | 5 分（優） |
|------|----------|-----------|-----------|
| 使用者體驗 | 錯誤訊息品質/help 文件/自動補全 | 錯誤訊息不明確、無 help、無補全 | 錯誤含修復建議、完整 help + 範例、shell 補全 |
| 可靠性 | 輸入驗證/錯誤處理/冪等操作 | 無輸入驗證、未處理邊界、操作不可重試 | 嚴格驗證、優雅降級、操作冪等可安全重試 |
| 跨平台性 | Win/Mac/Linux 相容與原生整合 | 僅支援單平台或路徑問題頻發 | 三平台完整支援、CI 環境偵測、原生路徑處理 |
| 效能 | 啟動速度/執行效率/資源佔用 | 冷啟動 > 1 秒、大檔案處理 OOM | 冷啟動 < 200ms、串流處理、記憶體恆定 |
| 可維護性 | 命令結構/測試覆蓋/擴充性 | 命令混亂、無測試、新增命令需改核心 | 命令樹清晰、整合測試完整、plugin 機制 |

### 權重規則

- **內部工具**：使用者體驗 20%、可靠性 30%、跨平台性 10%、效能 15%、可維護性 25%
- **開源工具**：使用者體驗 30%、可靠性 25%、跨平台性 20%、效能 10%、可維護性 15%
- **CI/CD 工具**：使用者體驗 10%、可靠性 35%、跨平台性 20%、效能 20%、可維護性 15%

## 🔍 自我審查

> 各 agent 在輸出前的自我檢查問題（疊加到內建預設之上）。

### 架構設計（architect）

1. 命令結構是否遵循「名詞-動詞」或一致的組織慣例？是否直覺易記？
2. 設定檔優先級（預設 → 全域 → 專案 → CLI 參數）是否有清晰的覆寫規則？
3. 所有操作是否在非互動模式下（CI 環境）可正常執行？是否能透過參數完全控制？

### 開發實施（developer）

1. 所有使用者輸入（參數、設定檔內容、stdin）是否做了驗證和邊界檢查？
2. 錯誤訊息是否包含具體問題描述和建議的修復操作？
3. stdout 和 stderr 是否正確分離？`--json` 輸出是否為合法 JSON？

### 代碼審查（code-reviewer）

1. 檔案路徑處理是否使用跨平台 API？是否考慮了空格和特殊字元？
2. 退出碼是否語義一致？是否有未捕獲的異常導致非預期退出碼？
3. 長時間操作是否有進度顯示？Ctrl+C 中斷時是否正確清理臨時資源？

## 📝 範例

> 各 agent 的 few-shot example，展示期望的輸出深度和格式。
> **啟用規則**：有定義時自動啟用 few-shot 模式。

### architect 範例

> **需求**：設計一個專案設定檔管理 CLI 工具，支援多環境設定（dev/staging/prod）、設定繼承與覆寫、敏感值加密儲存，可整合進 CI/CD pipeline。

**硬約束**：
- 冷啟動 ≤ 200ms，單次操作 ≤ 500ms
- 跨平台支援 Windows/macOS/Linux
- CI 環境下無互動操作，所有功能可透過參數控制
- 敏感值加密後可安全提交至 Git

**軟約束**：
- 設定格式支援 YAML 和 TOML
- 支援 `.env` 檔案匯入匯出
- 支援 shell 自動補全（bash/zsh/fish）
- 未來擴充：遠端設定中心同步

#### 方案 A：分層合併 + 本地加密

使用分層設定架構（`base.yaml` → `{env}.yaml` → 環境變數 → CLI 參數），敏感值使用 AES-256-GCM 加密後直接存在設定檔中，金鑰透過環境變數或 OS keyring 管理。

- 合併策略：deep merge，陣列可選替換或追加
- 加密：`config encrypt` / `config decrypt` 子命令，加密值以 `ENC(...)` 標記
- 輸出：`config get KEY` 取值、`config dump --env prod --json` 輸出完整設定

#### 方案 B：模板渲染 + 外部 Vault

使用 Go template 語法嵌入設定檔，變數值從環境變數或 HashiCorp Vault 讀取。執行時渲染成最終設定檔。

- 合併策略：模板渲染、無 deep merge、最終檔案即全部
- 加密：委託給 Vault，CLI 只負責存取
- 輸出：`config render --env prod` 生成完整設定檔

#### 結構化評估

| 維度 | 方案 A（分層合併 + 本地加密） | 方案 B（模板渲染 + Vault） |
|------|:--:|:--:|
| 使用者體驗 | 4 — 操作直觀、設定可讀性高 | 3 — 模板語法學習曲線、最終結果不直觀 |
| 可靠性 | 4 — 合併規則明確、加密自包含 | 3 — 依賴 Vault 可用性 |
| 跨平台性 | 5 — 純本地操作、無外部依賴 | 3 — Vault 需額外安裝和配置 |
| 效能 | 4 — YAML 解析 + merge 速度快 | 4 — 模板渲染速度快 |
| 可維護性 | 4 — 分層結構清晰、易於擴展 | 3 — 模板複雜後難以維護 |

**開源工具權重加權**：A = 4.15 / B = 3.10

#### 推薦

**推薦方案 A（分層合併 + 本地加密）**，理由：

1. 零外部依賴 — 不需要額外安裝 Vault，降低使用門檻
2. 分層 deep merge 符合開發者的直覺（base + overlay），設定檔保持可讀
3. `ENC(...)` 標記讓加密值可安全提交 Git，同時一目了然哪些是敏感值
4. CI 環境只需設定金鑰環境變數即可，無需建立 Vault 連線

**擴展預案**：未來需要遠端設定中心時，新增 `config sync` 子命令作為可選外掛，不影響本地操作流程。

#### 實施計劃

| 階段 | 任務 | 風險 | 產出 |
|------|------|------|------|
| P1 | CLI 框架 + 命令結構 + 設定檔解析 | 🟢 LOW | 基礎架構 + `config get/set/dump` |
| P2 | 分層合併引擎（deep merge + 環境變數覆寫） | 🟡 MEDIUM | 多環境合併功能 |
| P3 | AES-256-GCM 加密/解密 + 金鑰管理 | 🟡 MEDIUM | `config encrypt/decrypt` |
| P4 | `.env` 匯入匯出 + `--json` 輸出 | 🟢 LOW | 格式轉換功能 |
| P5 | Shell 自動補全 + 三平台 CI 測試 | 🟢 LOW | 安裝腳本 + CI pipeline |
