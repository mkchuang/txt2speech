# 專案 Prompt Engineering 指引

> **用途**：定義專案特定的 AI 行為指引，讓 agent 輸出更貼合領域需求
> **所有區塊皆為可選**：未填寫的區塊，agent 使用內建預設值
> **填寫時機**：`adf.design` 初始化專案時選擇 preset，或任何時候手動編輯
> **大小建議**：每個 agent 範例建議 50-100 行以內，避免佔用過多 context window
>
> **Preset**：`mobile` — 行動應用（常駐/市電場景）

---

## 🧠 角色補充

- 產品為視訊會議行動應用，部署於接市電的固定裝置（會議室平板等），不需考慮電池消耗
- 音視訊品質是第一優先，畫面流暢度和音訊清晰度直接影響會議體驗
- 網路環境不穩定是常態（Wi-Fi 切換、有線/無線切換），必須有完善的斷線恢復機制
- UI 回應性 < 100ms，視訊渲染幀率穩定 ≥ 25fps
- Crash-free rate 目標 ≥ 99.9%，任何 crash 都需要自動上報和分類
- 螢幕適配需覆蓋平板（7"-13"）和手機（4.7"-6.7"），支援橫豎屏切換
- 長時間運行場景常見（會議室全天開機），記憶體穩定性與資源回收至關重要
- 硬體存取（相機、麥克風、揚聲器）需優雅處理權限拒絕和裝置切換
- 與後端 SFU/MCU 的信令協定版本必須前後相容

## 📏 評估框架

> `/planner`（architect）比較方案時使用的評估維度和評分標準。
> **覆寫規則**：此處定義的維度**取代**內建預設（不是疊加）。

### 評估維度

| 維度 | 評估重點 | 1 分（差） | 5 分（優） |
|------|----------|-----------|-----------|
| 音視訊品質 | 畫面清晰度/音訊延遲/同步性 | 頻繁花屏或卡頓、音訊回音或延遲 > 300ms | 720p 穩定流暢、音訊延遲 < 150ms、唇音同步 |
| 網路韌性 | 斷線恢復/弱網適應/頻寬調節 | 斷線即掉會、無重連機制、弱網直接卡死 | 3 秒內自動重連、動態碼率調節、弱網降級平滑 |
| 使用者體驗 | UI 回應/佈局適配/操作直覺性 | 操作卡頓、佈局錯亂、常見操作步驟多 | 所有互動 < 100ms、自適應佈局、一鍵操作為主 |
| 穩定性 | Crash-free rate/記憶體洩漏/ANR | Crash rate > 1%、記憶體持續增長、頻繁 ANR | Crash-free ≥ 99.9%、記憶體穩定、零 ANR |
| 可維護性 | 程式碼結構/測試覆蓋/問題追蹤 | 音視訊邏輯與 UI 耦合、無 crash 上報 | 分層清晰、音視訊 SDK 獨立、完整 crash + 日誌上報 |

### 權重規則

- **Alpha 階段**：音視訊品質 30%、網路韌性 15%、使用者體驗 20%、穩定性 15%、可維護性 20%
- **Beta 階段**：音視訊品質 25%、網路韌性 25%、使用者體驗 20%、穩定性 20%、可維護性 10%
- **GA 階段**：音視訊品質 20%、網路韌性 25%、使用者體驗 15%、穩定性 30%、可維護性 10%

## 🔍 自我審查

> 各 agent 在輸出前的自我檢查問題（疊加到內建預設之上）。

### 架構設計（architect）

1. 音視訊 pipeline 與 UI 層是否完全解耦？SDK 是否可獨立測試和更新？
2. 網路中斷 → 恢復的完整狀態機是否定義？各狀態的超時和重試策略是否明確？
3. 與後端 SFU/MCU 的信令協定是否向後相容？版本協商機制是否完整？
4. 多人通話場景下的資源（記憶體、CPU、頻寬）上限是否有估算？

### 開發實施（developer）

1. 相機/麥克風權限被拒絕時是否有優雅的降級和引導？裝置熱插拔是否處理？
2. 音視訊資源（encoder、decoder、surface）是否在生命週期結束時確實釋放？
3. 弱網場景（丟包 10%+、RTT > 300ms）下是否有適當的降級策略？

### 代碼審查（code-reviewer）

1. 是否有在 UI 主執行緒執行的耗時操作（編解碼、網路請求、檔案 I/O）？
2. 多執行緒存取共享的音視訊狀態時，是否有適當的同步保護？
3. 記憶體密集型操作（影像處理、buffer 分配）是否有 pool/cache 機制避免頻繁 GC？

## 📝 範例

> 各 agent 的 few-shot example，展示期望的輸出深度和格式。
> **啟用規則**：有定義時自動啟用 few-shot 模式。

### architect 範例

> **需求**：在視訊會議行動應用中實作多人通話佈局功能，支援 2-25 人同時視訊，需在平板（10"）和手機（6"）上自動選擇最佳佈局，並支援 Wi-Fi 與有線網路之間的無縫切換。

**硬約束**：
- 支援 2-25 人同時視訊，大畫面 720p + 小畫面 180p
- 佈局切換動畫 ≤ 300ms，視訊渲染無黑屏或閃爍
- 網路切換（Wi-Fi → Ethernet → Wi-Fi）期間音訊中斷 ≤ 1 秒
- 25 人滿載時 CPU 佔用 ≤ 70%、記憶體增量 ≤ 200MB

**軟約束**：
- 支援手動置頂某個參與者的畫面
- 演講者偵測（active speaker）自動放大
- 未來擴充：虛擬背景、螢幕分享畫面

#### 方案 A：動態 Grid 佈局 + ICE Restart

佈局層使用自適應 Grid 演算法（根據人數和螢幕尺寸計算最佳格數），網路切換使用 ICE Restart 機制在不中斷 PeerConnection 的情況下切換候選路徑。

- 佈局引擎：根據人數分級（2-4: 等分、5-9: 重點+Grid、10-25: 分頁Grid）
- 渲染優化：可視範圍外的畫面暫停解碼，僅解碼可見的 feeds
- 網路切換：ICE Restart 重新收集候選、無需重建 PeerConnection

#### 方案 B：模板佈局 + 完整重連

佈局層使用預定義模板（2人/4人/9人/16人/25人），人數變化時切換模板。網路切換時完整重建連線（Leave + Rejoin）。

- 佈局引擎：5 套固定模板，人數不匹配時就近選擇
- 渲染優化：所有畫面統一解碼，模板切換時重新排列
- 網路切換：偵測到網路變化後完整重連，由 SFU 端恢復媒體流

#### 結構化評估

| 維度 | 方案 A（Grid + ICE Restart） | 方案 B（模板 + 重連） |
|------|:--:|:--:|
| 音視訊品質 | 5 — 可視區域才解碼、資源利用最佳 | 3 — 全解碼浪費資源、切換時有間斷 |
| 網路韌性 | 5 — ICE Restart 無縫切換、音訊不中斷 | 2 — 完整重連耗時 3-5 秒、音訊中斷明顯 |
| 使用者體驗 | 4 — 自適應佈局最佳但演算法複雜度高 | 3 — 模板有跳變感、人數不匹配時空間浪費 |
| 穩定性 | 3 — ICE Restart 邊界情況多需仔細處理 | 4 — 完整重連邏輯單純、狀態乾淨 |
| 可維護性 | 3 — Grid 演算法和 ICE Restart 需深度理解 | 4 — 模板直觀易修改、重連邏輯可復用 |

**GA 階段權重加權**：A = 4.10 / B = 3.05

#### 推薦

**推薦方案 A（動態 Grid + ICE Restart）**，理由：

1. 會議室場景頻繁切換網路（Wi-Fi 不穩切有線），ICE Restart 帶來的無縫切換是剛需
2. 25 人場景下僅解碼可見畫面可節省約 60% CPU，對硬體約束至關重要
3. 動態 Grid 在平板大螢幕上空間利用率遠優於固定模板
4. ICE Restart 的邊界情況可透過完善的狀態機和超時 fallback 至完整重連來緩解

**降級策略**：ICE Restart 超時 3 秒未成功，自動 fallback 至方案 B 的完整重連路徑。
